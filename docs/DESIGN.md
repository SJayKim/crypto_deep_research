# Design: deep_research — Multi-Agent Coin Deep-Research Agent

## Problem Statement

Build a multi-agent "deep research" system that pulls coin data and produces an
analysis report. The coin domain is deliberately **light**: the real goal is to
force a hands-on encounter with every 2026 core agent concept in one project:
orchestrator-worker, context isolation, distillation, layered memory, and the
MCP vs A2A protocol split. The coin domain is the excuse, not the point.

Stack: Python + LangGraph. Data sources: CoinGecko (price/meta), Binance
(realtime/orderbook), Upbit (KRW market). All three stay stubbed behind a tool
interface at first; live calls are a later swap.

## What Makes This Cool

A normal deep-research demo uses web search as the worker tool, so workers burn
tokens parsing messy HTML. Swapping web search for **structured coin APIs** makes
this a cheaper, sharper teaching rig: the expensive, interesting part of
multi-agent research (parallel subagents in isolated contexts) is exactly the
concept you want to feel, and structured JSON keeps each worker's context small.
You learn the architecture without drowning in domain noise.

## Constraints

- Learning fidelity is the north star, not analysis quality (chosen at D1).
- First milestone is ONE full vertical slice end to end, not a breadth tour (D2).
- Domain stays stubbed first; live exchange APIs are a later swap (premise 1).
- Python + LangGraph (decided prior session).

## Premises (all agreed at D3)

1. **Domain stays stubbed first.** The three exchanges hide behind one tool
   interface returning fixture data on day one. Live APIs are a later swap.
2. **The query must be genuinely decomposable, or orchestrator-worker is theater.**
   "Analyze BTC now" fans out to 4 independent workers: market/technicals,
   orderbook/liquidity, news/sentiment, on-chain. No decomposition, no orchestrator.
3. **The orchestrator never sees raw worker context.** Each worker runs in its own
   context, does its messy tool work there, and returns only a distilled summary.
   Context isolation (separate worker contexts, a topology property) and distillation
   (summarize-before-return, a transformation) are co-located at the worker boundary
   but are distinct properties. Verify each independently: you can have isolation
   without distillation (return raw, still isolated) and distillation without
   isolation. Building one and assuming you got both is the trap.
4. **MCP and A2A are different layers, kept separate.** MCP = agent-to-tool (coin
   data server). A2A = agent-to-agent (orchestrator to worker). Not conflated.
5. **Every memory layer needs a concrete trigger or it's decoration.** Working,
   episodic, and long-term each get a real read point and write point in the slice.

## Approaches Considered

### Approach A: Single-process LangGraph, in-proc workers (minimal viable)
One process, LangGraph Send fan-out, real MCP server, A2A as a typed pattern (not
wire). Effort M, Risk Low. Rejected as v1 because A2A stays simulated, and learning
the MCP/A2A split for real is a stated goal.

### Approach B: Service-separated, real MCP + real A2A over the wire (CHOSEN)
Orchestrator, each worker, and the MCP server are separate processes. A2A is a real
JSON-RPC protocol with Agent Cards. Context isolation is enforced by process
boundaries. Effort XL, Risk High. Chosen at D4 for maximum learning fidelity.

### Approach C: Dynamic hierarchical orchestrator + distillation/critic loop
In-proc but the orchestrator plans dynamically from query and memory, workers
recurse one level, a compressor produces tiered distillation and a verifier checks
claims. Effort L, Risk Med. Held as a later enhancement (its dynamic-planning and
tiered-distillation ideas graft onto B once B works).

## Recommended Approach (B, de-risked)

Real services, real protocols, **light domain**. The key move that keeps B from
stalling in infra: contracts first, then a working over-the-wire slice by
milestone 2, then breadth and depth.

### System shape

- **Orchestrator service** (LangGraph app): receives the query, reads memory, plans
  which workers to run, dispatches over A2A, collects distilled artifacts,
  synthesizes the report. Its context holds only distilled artifacts, never raw data.
- **Worker services** (4, each its own process, each an A2A server with an Agent Card):
  `market-worker`, `orderbook-worker`, `sentiment-worker`, `onchain-worker`. Each is
  a small LangGraph agent that calls the MCP server for data, reasons in its own
  context, and distills before responding over A2A.
- **Coin data MCP server** (one process): exposes tools `get_ohlcv(symbol, interval)`,
  `get_orderbook(symbol)`, `get_news(symbol)`, `get_onchain(symbol)`. Backed by a
  `DataSource` interface with a `FixtureSource` (day one), later `CoinGeckoSource` /
  `BinanceSource` / `UpbitSource`. Workers connect as MCP clients over **streamable
  HTTP** (not stdio: the server is its own process/container). Tools are stateless and
  read-only over fixtures, so concurrent access by 4 workers is trivially safe.
- **Layered memory** (library + DB, SQLite to start, not a service, to limit moving
  parts), each layer with an explicit read and write trigger:
  - working: per-run scratchpad. Write trigger = a worker records intermediate notes;
    read trigger = its distill node reads the scratchpad to build the artifact. Backed
    by the LangGraph checkpointer for durability (the checkpointer is the storage
    mechanism, not the memory itself).
  - episodic: past analyses per coin. Read trigger = run start retrieves the most
    recent run row keyed by symbol; write trigger = run end stores this run.
  - long-term: user watchlist + learned coin facts. Read trigger = the planner reads
    it to choose the worker set; write trigger = run end appends newly learned facts.

### Concept-to-component map (the heart of the design)

| 2026 concept       | Where it lives in B                                   | What makes it real                                              |
|--------------------|-------------------------------------------------------|----------------------------------------------------------------|
| Orchestrator-worker| Orchestrator plans + dispatches to worker services    | Workers are separate processes, dispatched over A2A            |
| Context isolation  | Each worker = own process + own LLM context           | Orchestrator literally cannot read worker raw context          |
| Distillation       | Each worker's distill node LLM-summarizes its own context | Artifact uses bounded fields (<=5 key points, capped lengths) via validators |
| Layered memory     | Working / episodic / long-term store with triggers    | Planner reads long-term to pick workers; episodic seeds context|
| MCP / A2A          | MCP = coin data tools; A2A = orchestrator to worker   | Two distinct protocols, two distinct boundaries, not conflated |

### Vertical slice: "analyze BTC now"

1. CLI entry hands the query to the orchestrator.
2. Orchestrator loads long-term memory (watchlist, BTC facts) and episodic memory
   (last BTC run). The planner decides the worker set (for BTC: all 4).
3. Orchestrator sends A2A task messages to the 4 worker services in parallel.
4. Each worker pulls data via MCP, reasons in its isolated context, distills to a
   typed `WorkerArtifact` with bounded fields: headline, <=5 key points, and evidence
   as self-contained quoted values (e.g. `{"metric":"RSI_14","value":71.2}`), NOT
   pointers into worker-local context the orchestrator cannot read. Returns over A2A.
5. The synthesizer merges the available artifacts (it never saw raw data) into a
   single synthesis report (headline, key points, evidence). Partial-failure policy:
   synthesize on >=1 successful artifact; mark any failed or timed-out dimension as
   unavailable rather than blocking the run. (Multi-tier distillation and a claim
   verifier are the Approach C enhancement, not part of B.)
6. Write episodic memory (this run), update long-term (new learned facts), return.

### Milestone sequence (de-risks the stall I flagged at D4)

- **M0 Contracts first.** Define the A2A message schema (task, artifact), the MCP
  tool schema, the `WorkerArtifact` Pydantic model, the memory interface. Also pin the
  wiring: static service URLs via env vars (one per worker + the MCP server), Agent
  Cards served at `/.well-known/agent.json`, MCP transport = streamable HTTP. No logic.
- **M1 One worker, direct.** market-worker + MCP server + fixture data, called
  directly (no A2A) to prove data to reason to distill.
- **M2 One worker over A2A.** Wrap market-worker as an A2A service; orchestrator
  calls it over A2A. First over-the-wire vertical slice. (Working slice by here.)
  Timebox the SDK: if the official `a2a` SDK fights you, fall back to minimal
  hand-rolled JSON-RPC + a static Agent Card. The protocol shape is the lesson,
  not the SDK.
- **M3 Fan-out.** Add a 2nd worker to prove parallel dispatch + synthesizer, then the
  remaining 2. Two workers already prove fan-out; stage the rest to limit domain noise.
- **M4 Memory.** Wire layered memory with real triggers (planner reads long-term;
  episodic seeded and written).
- **M5 Live data.** Swap one fixture source for a live API (CoinGecko) behind the
  same MCP tool, no agent code changes.

## Open Questions

- A2A implementation: official `a2a` Python SDK vs hand-rolled JSON-RPC.
  Recommend the official SDK so the protocol learning is real.
- Memory as service vs library: recommend library + DB (SQLite to start) to keep one
  less moving part. MCP and A2A are the concepts that need wire-realness, not memory.
- Worker reasoning: market/sentiment workers use LLM reasoning; orderbook can be
  mostly deterministic. Decide per worker.
- LLM provider: Anthropic Claude via the agents (ANTHROPIC_API_KEY already in
  `.env.example`).

## Success Criteria

- `analyze BTC now` (local processes during M1-M4, or `docker-compose up` at M5)
  returns a synthesis report.
- Orchestrator state provably contains only distilled artifacts (assert no raw OHLCV
  arrays ever enter orchestrator context).
- A2A calls are real: separate processes, verifiable in logs.
- MCP server serves all 4 tools; workers connect as MCP clients.
- Memory works: a second BTC run references the first (episodic), and the
  watchlist/facts change the plan (long-term).

## Distribution Plan

Local multi-service app. During M1-M4 run as plain local processes. At M5,
`docker-compose` brings up orchestrator + 4 workers + MCP server + DB as packaging.
Entry via CLI: `python -m deep_research "analyze BTC now"`. Quality gate: ruff + mypy
+ pytest via pre-commit (local). Defer GitHub Actions until a remote exists; CI on a
no-remote solo repo is gold-plating.

## Locked Decisions — Eng Review (2026-06-08)

Additive record from `/plan-eng-review`. The approved design above is unchanged;
these lock the implementation details, with two amendments to the milestone
sequence and success criteria noted at the bottom.

| # | Decision | Resolution |
|---|----------|------------|
| Scope | Worker count for the slice | All 4 workers stay in scope (no reduction). |
| A1 | A2A transport | Hand-roll minimal JSON-RPC 2.0 + static Agent Card (not the official SDK). |
| A2 | Distillation enforcement | `WorkerArtifact` Pydantic validators (≤5 key_points, capped lengths, evidence as typed `{metric, value}`) + flagship isolation test: feed 1000-row OHLCV, assert artifact bounded AND orchestrator state holds zero raw arrays. |
| A3 | Failure model | Per-worker 30s timeout (env-configurable) via `asyncio.gather`; zero-artifact path → `status=failed` report listing per-dimension reasons, CLI exits non-zero. |
| A4 | Memory DB topology | Each worker owns its own checkpointer DB; orchestrator solely owns episodic + long-term DB (single-writer-per-file). MCP server stateless. |
| C5 | Shared contracts | One shared `contracts/` Python package imported by all 6 services. |
| C6 | Worker duplication | Build market-worker concretely first; extract the shared Worker base harness only after the 2nd worker (rule of three). |
| T7/T7b | LLM in tests | Real Anthropic in worker-behavior + eval tests; deterministic stub in validator/timeout/MCP-down/zero+partial-artifact/memory/isolation-bound tests. (Recorded in CLAUDE.md.) |
| T8 | E2E transport | In-process ASGI on loopback ports for the routine E2E suite (real JSON-RPC/HTTP); M5 docker-compose is the single real-OS-process-boundary proof. |
| P9 | Fan-out mechanism | `asyncio.gather` over A2A. **NOT** LangGraph `Send` — `Send` dispatches in-graph nodes inside the orchestrator process and cannot cross the A2A boundary; using it silently collapses Approach B into Approach A. |

**Outside-voice refinements (independent reviewer, adopted):**
- **TENSION-A:** Held Approach B (process/wire spine is a deliberate learning target, premise 4; M1 already builds distillation pre-wire).
- **TENSION-B (milestone amendment):** Split memory — the long-term **READ** trigger (planner picks the worker set) ships with **M3 fan-out**, not M4; episodic/long-term **writes** + round-trip stay **M4**. Reason: the read-trigger shapes orchestration, so it must be exercised in the core slice, not bolted on at the end.
- **TENSION-C (success-criteria amendment):** The synthesis report carries explicit per-dimension coverage (e.g. `dimensions_ok`, `dimensions_unavailable`); the CLI surfaces it; a test asserts a 1-of-4 run is visibly marked partial. Extends A3's no-silent-failure rule from the zero case to the partial case.

**Worktree parallelization:** sequential spine M0 → MCP server → market-worker → A2A
spine → base-harness extract; then orderbook / sentiment / onchain run as 3 parallel
worktree lanes (each touches only its own `workers/<name>/`), with memory as a 4th
parallel lane. Keep the worker registry data-driven (env-var list of worker URLs) so
adding a worker never edits `orchestrator/`.
