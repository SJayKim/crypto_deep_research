## Inherits (from: [M3](./M3.md))

- **`orchestrator/app.py`** with run-start / run-end points — M4 plugs memory I/O here.
- **`orchestrator/planner.py`** reading long-term memory — M3 ran it on seeded fixtures;
  M4 makes the store self-populating so a real prior run feeds it.
- **`contracts/memory.py`** protocols (from M0) — `WorkingMemory`, `EpisodicMemory`,
  `LongTermMemory`, `RunRecord`. M4 provides the concrete implementations.

**Preconditions:** M3 acceptance criteria all pass (4-way fan-out; isolation flagship;
partial/zero/timeout policy; planner long-term READ against fixtures).

## Scope

Episodic write (run end stores `RunRecord`) + read (run start retrieves last run by
symbol); long-term write (run end appends learned facts); working memory via the
LangGraph checkpointer. DB topology per A4: each worker owns its checkpointer DB; the
orchestrator solely owns the episodic + long-term DB (single-writer-per-file); MCP
stateless.

## Acceptance criteria

1. A second `analyze BTC now` run reads the first via episodic `last_for("BTC")`
   and the run visibly references it (episodic round-trip).
2. A long-term fact written at run end changes the next run's plan (long-term
   WRITE → next-run READ closes the loop started in M3).
3. Each worker's checkpointer DB is a distinct file from the orchestrator's
   episodic/long-term DB (A4 verified by path assertion).
4. Concurrent workers writing their own checkpointer DBs never touch the
   orchestrator DB (single-writer-per-file holds).
5. Memory tests use deterministic stubs, not real LLM (T7b).

## Hands off (to: [M5](./M5.md))

- **A correct in-process slice** — all 4 dimensions, memory round-trip, partial/zero
  coverage, all green on loopback ASGI (T8). This is M5's precondition: only swap to live
  data + containers once the logic is proven cheap and in-process.
- **DB topology settled (A4)** — per-worker checkpointer DBs + orchestrator-owned
  episodic/long-term DB. M5's docker-compose mounts/maps these without changing the
  ownership model.

**Seam:** the slice is functionally complete and correct in-process. M5 is orthogonal —
it changes *where data comes from* and *how processes are packaged*, touching no agent or
memory logic.
